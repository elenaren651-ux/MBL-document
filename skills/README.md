# MBL Skills System

这套 `skills/` 是 MBL 制单项目的长期方法论资产库。

它的目标不是只服务一次性开发任务，而是把你在这个项目里反复验证过的产品判断、AI 抽取设计方法、业务规则和页面迁移经验沉淀下来，形成可复用、可持续优化的 skill 体系。

## 1. 为什么要在仓库里维护 skills

这个项目的价值不只在于做出一个制单系统，还在于沉淀一套围绕以下问题的稳定方法：

- OCR 和大模型如何协同做抽取
- 哪些字段可以自动通过，哪些字段必须人工审核
- 人机交互怎么设计，才能兼顾效率和准确率
- 如何评测抽取准确率，并持续优化
- 如何把复杂的船司规则和结果物逻辑结构化
- 如何把原型逐步迁成真实 React 项目

这些内容如果只停留在聊天记录里，会反复丢失；沉淀到仓库后，它会变成可持续迭代的项目资产和个人方法论资产。

## 2. 分层原则

当前 skill 体系分成两层：

### `skills/ai/`

这一层沉淀的是你作为 AI 产品经理的通用方法论，未来不仅可以服务 MBL 制单项目，也可以迁移到其他 AI 抽取、审核、自动化业务系统中。

### `skills/mbl/`

这一层沉淀的是当前项目特有的业务知识和交付方法，重点围绕：

- MBL 业务流程
- 船司差异规则
- 结果物模型
- 页面迁移

## 3. 当前 skill 地图

### AI 方法论层

#### `ai-extraction-workflow-designer`

定义 OCR、LLM、规则引擎、人工审核之间的整体协同方式。

#### `ai-extraction-eval-designer`

定义抽取准确率、字段级评测、样本集与误差分类体系。

#### `ai-human-in-the-loop-designer`

定义 AI 抽取后的人工审核台、人机分工和审核优先级设计。

#### `ai-field-confidence-designer`

定义字段级置信度、多来源候选值、自动通过与人工介入触发条件。

#### `ai-ocr-llm-orchestrator`

定义不同文档类型、不同字段类型下，OCR 与 LLM 的编排策略。

#### `ai-extraction-error-taxonomy`

定义抽取错误分类、根因分析和优化优先级。

### MBL 项目层

#### `mbl-page-migrator`

定义如何把现有原型页面迁成 React 页面，并保留业务入口和状态结构。

#### `mbl-carrier-rules`

沉淀船司差异化规则，以及这些规则对页面、流程和数据模型的影响。

#### `mbl-field-audit-designer`

围绕 MBL 制单详情页中的字段审核、冲突确认、缺失提示设计交互方案。

#### `mbl-result-object-modeler`

沉淀预览件、草稿件、VC件、提交记录等结果物模型及其页面结构。

#### `mbl-prd-writer`

把口头需求、飞书调研、原型变更整理成统一 PRD 和验收标准。

## 4. 优先级建议

建议优先持续打磨这 6 个：

1. `ai-extraction-workflow-designer`
2. `ai-extraction-eval-designer`
3. `ai-human-in-the-loop-designer`
4. `mbl-carrier-rules`
5. `mbl-page-migrator`
6. `mbl-result-object-modeler`

原因是这几项最直接决定：

- AI 能力方案是否靠谱
- 前端和产品是否能走通主链路
- 业务规则是否被正确固化

## 5. 每个 skill 的标准结构

每个 skill 目录下默认包含一个 `SKILL.md`，建议逐步统一成下面这些部分：

1. 目标
2. 适用场景
3. 输入
4. 输出
5. 工作流程
6. 关键判断原则
7. 和 MBL 项目的关联
8. 当前版本已知不足
9. 下一轮优化方向

第一版不要求每个 skill 都写得很长，但要求：

- 能真实指导一次具体任务
- 能明确边界
- 能记录当前认知阶段

## 6. 自我优化机制

这套 skill 允许随着业务理解加深持续迭代。推荐按下面的节奏维护：

### 6.1 迭代触发条件

当出现以下情况时，就应该回写对应 skill：

- 新增了一个重要业务规则
- 原有规则在真实场景里被推翻
- 某种抽取错误反复出现
- 人工审核环节出现明显低效或误判
- 某个页面迁移后发现原型理解不完整
- 评测口径发生变化

### 6.2 迭代内容

每次更新 skill 时，优先补这些内容：

- 新增的判断条件
- 更明确的输入输出
- 反例或异常场景
- 典型错误案例
- 新的验收标准

### 6.3 迭代来源

skill 的更新素材优先来自：

- 原型迭代
- PRD 规则补充
- 真实页面迁移过程
- 抽取错误复盘
- 人工审核交互复盘
- 准确率评测结果

## 7. 推荐维护方式

建议把 `skills/` 当作一个“持续生长的业务知识层”：

- 做完一次重要页面迁移，就补 `mbl-page-migrator`
- 新增一个船司规则，就补 `mbl-carrier-rules`
- 调整抽取链路，就补 `ai-extraction-workflow-designer`
- 建立评测口径，就补 `ai-extraction-eval-designer`

这样这套 skill 不会停留在空模板，而会和项目一起长。

## 8. 当前阶段结论

对 MBL 制单项目来说，最值得优先投入的不是“通用开发 skill”，而是：

- AI 抽取工作流设计
- 人机审核设计
- 抽取准确率评测
- 船司规则沉淀

这些才是项目未来最有壁垒、也最值得反复优化的核心能力。
