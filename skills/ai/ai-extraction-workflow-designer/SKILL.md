# ai-extraction-workflow-designer

## 1. 目标

为 MBL 制单项目设计一套真正可执行的抽取工作流，把“邮件接入 - 文档分流 - OCR/LLM 抽取 - 字段候选值生成 - 人工审核 - 进入后续提交链路”串成一条可以持续优化的标准流程。

这个 skill 的重点不是单次抽取效果，而是设计一套长期可扩展、可评测、可回溯的抽取系统。

## 2. 适用场景

- 设计 MBL 制单的智能提取主链路
- 规划邮件正文、PDF、图片、Excel、Word 等不同输入的处理方式
- 设计 OCR、LLM、规则引擎、人工审核之间的职责分工
- 明确抽取失败、字段冲突、低置信度场景的处理策略
- 为后续准确率评测和审核台设计提供统一输入格式

## 3. 这个 skill 在 MBL 项目中要解决什么问题

MBL 制单的真实业务难点不是“从文档里提字段”这么简单，而是下面这几个问题同时存在：

- 客户补料来自邮件正文和多个附件，格式不统一
- 同一字段可能有多个来源，而且彼此不一致
- OCR 负责看清，LLM 负责理解，但两者都可能出错
- 不是所有字段都可以自动通过
- 人工审核必须高效，否则系统只是把人工复制粘贴换成了人工逐字段复核

因此，这个 skill 的产出必须覆盖：

1. 输入如何分流
2. 哪类字段怎么抽
3. 抽完之后如何生成候选值
4. 哪些情况进入人工审核
5. 什么时候可以进入后续提交链路

## 4. 默认输入

当用这个 skill 设计 MBL 抽取流程时，默认输入包括：

- 业务目标
- 船司范围
- 字段清单
- 输入文档样本
- 当前异常类型
- 是否已有人工审核页面
- 是否已有字段来源与候选值模型

如果信息不全，默认按当前仓库已知背景处理：

- 输入来源是客户补料邮件
- 输入载体包括邮件正文和附件
- 抽取结果要进入字段审核页
- MVP 阶段允许后续提交链路未完全打通

## 5. 默认输出

使用这个 skill 时，默认应产出以下内容：

1. MBL 抽取流程分层图
2. 文档类型分流策略
3. OCR/LLM/规则/人工审核职责划分
4. 字段候选值生成逻辑
5. 人工审核触发规则
6. 失败兜底策略
7. 给评测体系和审核台的标准化输出接口

## 6. MBL 推荐工作流总览

### 6.1 主流程

MBL 制单项目推荐的抽取工作流如下：

1. 邮件接入
2. 邮件级预处理
3. 文档分流
4. OCR / 结构化读取 / 文本抽取
5. 字段级抽取
6. 规则标准化与候选值生成
7. 冲突、缺失、异常判断
8. 人工审核
9. 生成最终字段集
10. 进入提交船司、提交预览件、结果物回收等后续链路

### 6.2 设计原则

- 邮件级问题和字段级问题分开处理
- 识别层、理解层、决策层分开处理
- 原始文本、抽取结果、标准化结果都要保留
- 最终值不是直接由模型“拍板”，而是由候选值决策体系得出
- 人工审核只介入高风险、高冲突、高不确定度场景

## 7. 分层设计

### 7.1 第 1 层：邮件接入层

职责：

- 监听货代邮箱
- 拉取邮件元数据、正文、附件
- 生成邮件记录和原始输入快照

最少要保存：

- 邮件 ID
- 发件人
- 收件时间
- 邮件主题
- 邮件正文原文
- 附件列表
- 线程关系

这一层不要做：

- 直接给出最终字段值
- 在邮件级直接判定所有业务结论

### 7.2 第 2 层：邮件级预处理层

职责：

- 清洗邮件正文
- 去除签名、引用回复、无关抄送说明
- 做附件基础识别
- 尝试识别提单号、船司、客户身份等“工单归档线索”

产出：

- 清洗后的正文文本
- 附件类型标签
- 邮件级候选线索

注意：

- 提单号、船司、船司账号在这里可以先做粗识别
- 即使识别异常，也应尽量生成记录，不要直接丢弃

### 7.3 第 3 层：文档分流层

职责：

- 按输入类型选择后续处理策略

MBL 项目建议先按下面的类型分流：

| 文档类型 | 典型来源 | 推荐处理方式 |
|---|---|---|
| email_body | 邮件正文 | 文本清洗 + LLM 抽取 + 规则识别 |
| pdf_text | 可选中文本 PDF | 直接抽文本 + LLM 抽取 |
| pdf_scan | 扫描版 PDF | OCR + 版面理解 + LLM 抽取 |
| image | JPG/PNG/截图 | OCR + LLM 抽取 |
| excel | xlsx/xls/csv | 结构化读取 + 规则映射 + LLM 补理解 |
| word | doc/docx | 文本抽取 + LLM 抽取 |
| unknown | 无法识别格式 | 保留原件 + 人工补判 |

关键原则：

- 不要把所有附件统一当成 OCR 图片
- Excel、CSV 这类结构化输入优先走结构化读取
- 可提纯文本的 PDF 不要先 OCR 再 LLM，先走文本路径

### 7.4 第 4 层：识别层

职责：

- 把附件或正文尽可能转成可供抽取的文本或结构化片段

推荐职责划分：

- OCR：负责看清文字、表格、版面块
- Parser：负责读取 Excel、Word、可提文本 PDF
- Cleaner：负责清洗噪音、分段、切块

这一层产物必须保留：

- 原始文本
- OCR 文本
- 文档分块
- 表格区域
- 页码或来源位置

因为后面字段冲突、人工追溯、准确率评测都需要这些中间产物。

### 7.5 第 5 层：字段级抽取层

职责：

- 从识别层产物中抽取 MBL 制单所需字段

建议把字段分成 4 类：

#### A. 锚点字段

用于建单、归档、分单、识别主任务的字段。

例如：

- 提单号
- 船司
- 船司账号

#### B. 高风险核心字段

错误会直接导致提交失败、改单或业务损失。

例如：

- shipper
- consignee
- notify party
- port of loading
- port of discharge
- place of receipt
- place of delivery
- package / gross weight / measurement
- container / seal / marks

#### C. 派生字段

可由多个基础字段或业务规则推导。

例如：

- 是否满足提交条件
- 某些默认值
- 某些船司规则字段

#### D. 低风险辅助字段

主要用于展示、追溯、补充说明。

例如：

- 邮件主题映射字段
- 备注信息
- 客户补充说明

关键原则：

- 锚点字段和高风险核心字段要优先抽
- 派生字段不要直接让模型拍脑袋生成，优先由规则层推导

### 7.6 第 6 层：候选值生成层

职责：

- 将不同来源抽取到的字段值组织成候选值池

每个字段候选值至少保留：

- `fieldKey`
- `value`
- `normalizedValue`
- `sourceType`
- `sourceId`
- `sourceLabel`
- `extractedAt`
- `evidence`
- `confidence`

推荐来源类型：

- `email_body`
- `attachment`
- `customer_link`
- `manual_save`
- `system_default`

关键原则：

- 抽取结果先进入候选池，再决定最终值
- 候选池是审核台和评测体系的共同底座

### 7.7 第 7 层：规则标准化与决策层

职责：

- 对抽取结果做标准化、去噪、候选值合并、冲突判断

至少要做这些动作：

1. 标准化
2. 空值归一
3. 多来源合并
4. 冲突判断
5. 自动通过判断
6. 缺失判断

标准化示例：

- 去首尾空格
- 合并连续空格
- 统一大小写策略
- 港口名、提单号、箱号走字段级标准化

### 7.8 第 8 层：人工审核层

职责：

- 承接冲突值、低置信度值、缺失值和异常值

必须进入人工审核的典型场景：

- 锚点字段缺失
- 锚点字段有多个候选值且不一致
- 高风险核心字段置信度不足
- 客户链接回填值与当前值冲突
- 新邮件重新解析后与已确认值冲突
- 船司规则判断无法自动决策

可以考虑自动通过的场景：

- 同一字段多来源一致
- 字段为低风险辅助字段且来源质量高
- 结构化附件字段与正文字段一致

### 7.9 第 9 层：最终字段集输出层

职责：

- 给详情页右侧表单和后续提交链路输出统一结构

输出建议分成三部分：

1. `finalFields`
2. `candidateValues`
3. `reviewTasks`

其中：

- `finalFields` 是当前展示值
- `candidateValues` 用于来源追溯与候选引用
- `reviewTasks` 用于驱动人工审核动作

## 8. MBL 推荐的 OCR / LLM / 规则职责划分

### OCR 负责

- 图片、扫描 PDF 的文本识别
- 表格块识别
- 位置信息保留

### LLM 负责

- 从正文和识别结果中理解语义
- 识别字段语义映射
- 从多段文本中找目标字段
- 输出结构化候选字段

### 规则引擎负责

- 标准化
- 候选值合并
- 冲突定义
- 自动通过 / 必审判断
- 船司规则和客户规则套用

### 人工审核负责

- 冲突处理
- 缺失补齐
- 高风险字段确认
- 异常工单推进

## 9. MBL 推荐的文档处理策略

### 9.1 邮件正文

推荐策略：

- 清洗引用回复和签名
- 保留段落结构
- 先抽锚点字段，再抽高风险字段
- 正文中的自由文本更适合由 LLM 做语义抽取

### 9.2 Excel / CSV

推荐策略：

- 优先结构化读取，不先 OCR
- 通过列头、表头和上下文做字段映射
- 对异常列名再用 LLM 辅助理解

### 9.3 文本型 PDF

推荐策略：

- 直接抽文本
- 保留页码和段落块
- 让 LLM 在页码上下文中抽字段

### 9.4 扫描 PDF / 图片

推荐策略：

- 先 OCR
- 再按版面块切分
- 对表格和自由文本分别处理
- 最后由 LLM 汇总成字段候选值

### 9.5 Word

推荐策略：

- 优先抽文本
- 保留表格和段落层级
- 再做字段理解

## 10. 推荐的人机分工

### 系统自动完成

- 邮件接入
- 文档分流
- 文本识别
- 初始候选值生成
- 标准化
- 冲突、缺失、异常初判

### 人工必须处理

- 多来源冲突字段
- 锚点字段异常
- 船司账号异常
- 客户链接与当前值冲突
- 船司规则无法自动决策场景

## 11. 推荐的人工审核触发规则

以下任一条件满足，就生成审核任务：

1. 提单号缺失或多个提单号
2. 船司无法唯一判断
3. 船司账号异常
4. 同一高风险字段存在多个不一致候选值
5. 高风险字段为空
6. 新输入覆盖已确认值
7. 规则校验失败

审核任务建议包含：

- 问题类型
- 涉及字段
- 候选值摘要
- 建议动作
- 来源入口

## 12. 推荐的失败兜底策略

### 12.1 邮件级失败

如果邮件无法完整解析：

- 仍生成工单
- 标记为“待处理”或异常
- 保留原邮件和附件
- 允许人工指定提单号、船司账号等锚点信息

### 12.2 文档级失败

如果某个附件解析失败：

- 记录失败原因
- 不影响其他附件继续抽取
- 将失败附件暴露给审核台

### 12.3 字段级失败

如果字段无法可靠抽取：

- 保持空值
- 标记缺失
- 生成审核任务

### 12.4 决策级失败

如果候选值存在冲突且系统无法决策：

- 不自动覆盖最终值
- 标记冲突
- 等待人工引用或修改

## 13. 推荐的标准化输出结构

在 MBL 项目中，抽取层建议至少输出这 4 类对象：

### `mailRecord`

- 邮件元数据
- 原文
- 附件列表

### `documentArtifacts`

- 文档类型
- OCR 文本 / 解析文本
- 页码 / 片段 / 坐标

### `fieldCandidates`

- 字段候选值
- 来源
- 置信度
- 标准化值
- 证据片段

### `reviewTasks`

- 冲突
- 缺失
- 锚点异常
- 规则异常

## 14. 这个 skill 在 MBL 里的推荐落地顺序

如果按 MVP 推进，建议这样落地：

### 阶段 1

- 跑通邮件接入
- 跑通正文和常见附件识别
- 生成候选值池

### 阶段 2

- 跑通字段审核页
- 支持候选值引用、人工修正、确认状态

### 阶段 3

- 建立自动通过和人工审核触发规则
- 建立 review task 机制

### 阶段 4

- 补船司规则
- 补准确率评测闭环
- 补持续优化机制

## 15. 使用这个 skill 时的强制检查项

当用这个 skill 产出方案时，必须检查：

1. 是否区分了邮件级、文档级、字段级问题
2. 是否保留了候选值池，而不是直接给最终值
3. 是否明确了 OCR、LLM、规则、人工的职责边界
4. 是否定义了人工审核触发条件
5. 是否给评测体系保留了中间产物
6. 是否支持后续船司规则扩展

## 17. 当前版本适用于 MBL 的结论

对当前 MBL 制单项目，最推荐的抽取方案不是“单模型直接输出最终字段”，而是：

`邮件接入 -> 文档分流 -> OCR/解析 -> LLM 抽取候选值 -> 规则标准化 -> 候选值池 -> 人工审核 -> 最终字段集`

这是当前最适合你这个项目的基础框架，因为它同时兼顾了：

- 业务准确率要求高
- 输入格式不稳定
- 字段来源多
- 船司规则复杂
- 后续要做准确率评测和持续优化
