# ai-extraction-eval-designer

## 1. 目标

为 MBL 制单项目建立一套真正可执行的字段准确率评测框架，用来持续衡量：

- 抽取层是否可靠
- 候选值层是否充分
- 冲突识别是否有效
- 人工审核是否真正提升了提交前准确率

这个 skill 不只是“算一个 accuracy”，而是为整个 MBL 抽取工作流建立统一的评测语言和优化闭环。

## 2. 适用场景

- 设计 MBL 字段级 benchmark
- 比较不同 OCR、LLM、规则方案的效果
- 评估人工审核前后质量差异
- 分析不同文档类型、不同船司、不同客户格式下的表现
- 为优化优先级提供量化依据

## 3. 这个 skill 在 MBL 项目中要解决什么问题

MBL 制单场景下，单纯说“模型准不准”没有意义，因为真实业务至少有 4 层质量问题：

1. 文档有没有被正确读出来
2. 字段有没有被正确抽出来
3. 候选值和冲突有没有被正确组织出来
4. 最终提交前的字段值是不是正确

如果不把这几层拆开，团队就会出现这些问题：

- 不知道错在 OCR、LLM、规则还是审核
- 只看整体准确率，看不到关键字段问题
- 优化之后无法判断是“真的变好”还是“样本变简单了”
- 无法回答“人工审核到底帮了多少”

所以这套评测框架必须同时服务：

- 模型优化
- 产品交互优化
- 审核效率优化
- 业务准确率追踪

## 4. 默认输入

使用这个 skill 时，默认输入包括：

- 字段清单
- 抽取工作流输出
- 样本文档集
- ground truth 定义
- 当前候选值结构
- 当前审核结果

如果信息不完整，默认按当前仓库已有前提处理：

- 输入来源为客户补料邮件、正文和附件
- 抽取结果进入候选值池
- 人工审核是主链路的一部分
- 提单号、船司、船司账号和详情页核心字段为优先评测对象

## 5. 默认输出

使用这个 skill 时，默认应产出：

1. 指标体系
2. 字段分层评测口径
3. 样本集设计方法
4. ground truth 标注原则
5. 错误分类方式
6. 优化优先级判断规则
7. 与 `ai-extraction-workflow-designer` 配套的数据接口要求

## 6. MBL 评测框架总览

推荐把 MBL 抽取质量拆成 4 层评测：

1. 输入识别层
2. 字段候选值层
3. 最终字段值层
4. 审核与流程层

### 6.1 输入识别层

看文档有没有被正确读出来。

### 6.2 字段候选值层

看系统有没有把正确答案放进候选池。

### 6.3 最终字段值层

看最终展示给操作员、用于提交的字段值是否正确。

### 6.4 审核与流程层

看人工审核是否有效，流程是否真的降低了提交风险。

## 7. 评测对象分层

### 7.1 锚点字段

这类字段决定工单能不能正确归档和推进，优先级最高。

包括但不限于：

- 提单号
- 船司
- 船司账号

### 7.2 高风险核心字段

这类字段会直接影响提交成功率和业务风险。

包括但不限于：

- shipper
- consignee
- notify party
- port of loading
- port of discharge
- place of receipt
- place of delivery
- package
- gross weight
- measurement
- container
- seal
- marks

### 7.3 派生字段

这类字段通常来自规则推导，而不是单次抽取。

例如：

- 是否满足提交条件
- 某些船司默认值
- 某些规则覆盖值

### 7.4 低风险辅助字段

对提交影响较小，但对追溯和体验有帮助。

例如：

- 邮件备注映射
- 客户补充说明
- 页面辅助展示字段

## 8. MBL 推荐指标体系

### 8.1 字段级核心指标

#### `field_accuracy`

字段最终值与 ground truth 完全一致的比例。

适合：

- 最终字段值评测
- 提交前准确率追踪

#### `field_recall`

该字段应有值时，系统成功给出正确值或正确候选值的比例。

适合：

- 看系统是否漏抽

#### `field_precision`

系统给出值时，其中正确的比例。

适合：

- 看系统是否乱抽

#### `candidate_hit_rate`

正确答案是否至少出现在候选值池中。

这个指标在 MBL 特别重要，因为它直接回答：

“即使模型没直接选对，系统有没有把正确答案准备给审核员。”

#### `conflict_detection_recall`

对于真实存在冲突的字段，系统成功标记为冲突的比例。

#### `missing_detection_recall`

对于真实缺失的字段，系统成功标记为缺失的比例。

#### `auto_pass_precision`

所有被系统自动通过的字段中，真正正确的比例。

这个指标非常关键，因为它决定能不能放心减少人工审核负担。

### 8.2 审核相关指标

#### `pre_review_accuracy`

人工审核前，最终候选决策值的准确率。

#### `post_review_accuracy`

人工审核后，用于后续提交的最终值准确率。

#### `review_gain`

`post_review_accuracy - pre_review_accuracy`

这个指标回答：

“人工审核到底帮系统提升了多少。”

#### `review_load`

平均每单需要人工处理的字段数。

#### `review_focus_rate`

人工处理字段中，高风险字段占比。

这个指标可以用来判断审核资源有没有被浪费在低价值字段上。

### 8.3 流程级指标

#### `document_parse_success_rate`

文档识别成功率。

#### `work_order_creation_success_rate`

邮件进入系统后，成功形成可处理工单的比例。

#### `submission_ready_rate`

工单在审核后达到可提交状态的比例。

#### `critical_error_escape_rate`

高风险字段错误未被识别、仍进入后续链路的比例。

这个指标应尽量接近 0。

## 9. 字段评分口径

### 9.1 强匹配字段

这些字段要求几乎完全一致：

- 提单号
- 船司账号
- 箱号
- seal
- 港口代码类字段

评测时建议：

- 标准化后仍要求严格匹配

### 9.2 规范化匹配字段

这些字段允许经过标准化后匹配：

- 港口名
- 公司名称
- 地址
- 通知人信息

评测时建议：

- 允许空格、大小写、部分常见缩写差异被标准化吸收

### 9.3 结构化匹配字段

这些字段可能由多个子字段组成：

- container 行
- package / gross weight / measurement 组合字段

评测时建议：

- 不只看整体字符串
- 要支持子字段或行级比对

## 10. Ground Truth 设计原则

### 10.1 Ground truth 以“业务正确值”为准

不是以原文出现频率最高的值为准，也不是以模型最常输出的值为准，而是以：

- 最终经业务确认的正确值
- 或明确规则下的合法提交值

作为标准答案。

### 10.2 Ground truth 要区分 3 种状态

每个字段不只要标正确答案，还要标注状态：

- `present_correct`
- `missing_expected`
- `ambiguous_requires_review`

因为有些字段不是“抽对就好”，而是本来就应该进入人工审核。

### 10.3 Ground truth 要保留来源证据

每个标注字段建议同时保存：

- 正确值
- 对应来源文档
- 对应证据片段
- 标注备注

这样后续复盘时不会只剩一个“答案”，却不知道答案从哪来。

## 11. 样本集设计

### 11.1 样本不能只抽“正常单”

MBL 评测样本必须覆盖：

- 正常邮件正文
- 多附件场景
- 扫描件质量差场景
- Excel 补料场景
- 多来源冲突场景
- 缺失字段场景
- 船司账号异常场景
- 多提单号或提单号异常场景

### 11.2 推荐样本分层

建议至少按下面几维分层：

#### 按输入类型分层

- 仅正文
- 正文 + PDF
- 正文 + 图片
- 正文 + Excel
- 多附件混合

#### 按文档质量分层

- 高质量文本
- 中等质量扫描件
- 低质量扫描件

#### 按业务复杂度分层

- 单一来源一致
- 多来源一致
- 多来源冲突
- 缺失信息
- 异常单

#### 按船司分层

- MSK
- COSCO
- 其他后续接入船司

### 11.3 推荐样本集版本化

样本集不要长期混用一锅，建议做版本管理：

- `benchmark_v1`
- `benchmark_v2`
- `benchmark_carrier_msk_v1`

## 12. 候选值层评测

这一层是 MBL 项目和很多普通抽取项目的关键差异。

除了看最终值对不对，还必须看：

### 12.1 正确答案是否进入候选池

如果正确答案不在候选池，审核员再强也没法快速选对。

### 12.2 候选池是否过载

如果候选池塞了太多噪音值，审核效率会明显下降。

建议观察：

- 平均每字段候选值数
- 高风险字段平均候选值数
- 冲突字段候选值噪音比例

### 12.3 候选值排序是否合理

除了“有没有”，还要看“排得靠不靠前”。

建议增加：

- `top1_candidate_hit_rate`
- `top3_candidate_hit_rate`

## 13. 冲突与缺失评测

### 13.1 冲突识别

需要分别评估：

- 实际冲突字段里，被系统成功标红的比例
- 系统标红字段里，真实有冲突的比例

可拆成：

- `conflict_detection_recall`
- `conflict_detection_precision`

### 13.2 缺失识别

需要分别评估：

- 真实缺失是否被发现
- 非缺失字段是否被误报

可拆成：

- `missing_detection_recall`
- `missing_detection_precision`

## 14. 审核增益评测

MBL 项目不能只评模型，还必须评“人机协同结果”。

建议至少做这 3 组对比：

### 14.1 抽取后 vs 审核后

看人工审核整体增益。

### 14.2 高风险字段 vs 低风险字段

看审核资源是否用在关键处。

### 14.3 自动通过字段 vs 人工必审字段

看自动通过策略是否安全。

## 15. 错误分类框架

评测输出必须能归因。推荐最少分成这 6 类：

1. `ocr_error`
2. `parser_error`
3. `llm_extraction_error`
4. `normalization_error`
5. `candidate_merge_error`
6. `review_or_decision_error`

如果要更细，可以继续拆：

- 漏抽
- 错抽
- 值截断
- 单位错误
- 多行错位
- 默认值覆盖错误

## 16. 优化优先级判断规则

不要看到错误就平均修。建议按下面规则排序：

1. 锚点字段错误优先级最高
2. 会进入提交链路的高风险字段错误优先级高于展示字段错误
3. 自动通过字段中的错误优先级高于人工已拦截错误
4. 高频错误优先级高于低频边缘错误
5. 跨船司共性错误优先级高于单船司局部错误

## 17. 与工作流 skill 的配套关系

这个 skill 默认依赖 [ai-extraction-workflow-designer](/Users/renwenna/Desktop/MBL%20document%20make/MBL-document/skills/ai/ai-extraction-workflow-designer/SKILL.md) 的输出。

两者对应关系如下：

| 工作流层 | 对应评测对象 |
|---|---|
| 邮件接入层 | work order creation success |
| 文档分流层 | parse success by doc type |
| 识别层 | OCR / parser quality |
| 字段抽取层 | field recall / precision |
| 候选值层 | candidate hit rate |
| 决策层 | auto pass precision / conflict recall |
| 人工审核层 | review gain / review load |
| 最终字段集 | post review accuracy |

## 18. 评测结果建议的呈现方式

1. 按字段
2. 按文档类型
3. 按船司
4. 按审核前后

如果需要更细，还可以加：

- 按客户
- 按异常类型
- 按模型版本

## 19. 这个 skill 的标准输出模板

当用这个 skill 产出方案时，建议至少输出：

1. 字段清单与分层
2. 指标定义表
3. ground truth 标注规则
4. 样本集分层方案
5. 错误分类树
6. 优化优先级列表

## 20. 当前阶段对 MBL 的结论

对 MBL 制单项目，最值得优先建立的不是“整体 accuracy”，而是下面这几个核心指标：

1. 锚点字段准确率
2. 高风险字段最终准确率
3. 候选值命中率
4. 冲突识别召回率
5. 人工审核增益
6. 自动通过准确率

这 6 个指标足够支撑你在 MVP 阶段判断：

- 系统能不能用
- 哪一层最拖后腿
- 下一轮该优先优化模型、规则还是审核交互
