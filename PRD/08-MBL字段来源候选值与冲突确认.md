# MBL制单-字段来源候选值与冲突确认

## 1. Summary

本需求用于定义 MBL 制单详情页中“字段来源、候选值、确认状态、取值冲突、变更记录”的交互规则。目标是在大模型抽取不稳定、同一字段多来源取值可能不一致的情况下，让操作员可以快速判断字段是否可信、是否需要处理，并能追溯字段值从哪里来、被谁改过。

## 2. Contacts

| 角色 | 关注点 | 备注 |
|---|---|---|
| 操作员 | 快速审核字段是否正确 | 需要减少逐字段人工比对成本 |
| 业务方 | 提单提交准确率 | 提单字段不能出错，错误会造成改单、罚金或客户投诉 |
| 产品 | 设计字段级确认和冲突处理规则 | 需要统一页面交互语义 |
| 研发 | 实现来源记录、候选值、冲突判断和保存日志 | 需要明确数据结构和状态流转 |
| 测试 | 覆盖多来源冲突、确认、引用、保存记录 | 需要明确验收场景 |

## 3. Background

海运货代操作员过去需要接收客户补料，人工审核船司截单所需信息。如果资料缺失或错误，操作员再联系客户补充。客户或操作员再把资料复制粘贴到船公司官网，提交提单截单信息。

MBL 制单系统希望把这条链路线上化和智能化：

- 监听货代邮箱，获取客户发来的提单补料邮件。
- 从邮件正文和附件中智能提取提单字段。
- 操作员在平台中审核提取结果。
- 字段满足船司提交要求后，平台提交给船司官网或通道。
- 对缺失或有问题的信息，可通过链接或邮件让客户补充。

但是提单对准确率要求极高，不能有任何字段差错。大模型和 OCR 抽取存在不稳定性。一个字段可能同时从附件、邮件正文、客户链接、人工修改中得到多个值。多个来源的值不一致时，系统不能直接采用其中一个值，应明确提示操作员人工确认。

因此需要设计字段级的来源候选值、冲突提示、确认状态和变更记录。

## 4. Objective

目标是让操作员能在制单详情页中快速完成字段审核，并在出错时能追溯字段值来源。

关键结果：

- 操作员可以看到每个字段是否已确认。
- 操作员可以点击字段前 icon 查看该字段的所有候选值和变更记录。
- 当多个来源的字段值不一致时，字段能明确标红提示冲突。
- 操作员可以从候选值中引用某个值填入字段。
- 人工修改并保存后的值也进入字段来源记录。
- 通过客户链接回填的字段值可以进入候选值，并按冲突规则处理。
- 字段审核完成后，系统可以清楚知道哪些字段仍未确认、哪些字段存在冲突。

## 5. Market Segment(s)

本需求面向处理 MBL 制单的货代操作员、业务管理人员、平台测试和研发。

适用范围：

- 页面：MBL 制单详情页、待处理详情页、待提交船司详情页、提交前校验页。
- 数据对象：所有由系统抽取、客户回填或人工维护的提单字段。
- 来源类型：邮件正文、附件、客户链接、人工修改保存记录、系统推荐默认值。
- 船司：先适用于 MSK 当前制单流程，后续可扩展到其他船司。

暂不纳入范围：

- 自动决定哪个冲突值一定正确。
- 自动识别往来邮件属于哪个提单号。
- 船司官网提交后的截图字段反向解析。
- 客户规则配置的完整产品化，仅保留字段来源入口能力。

## 6. Value Proposition(s)

对操作员：

- 能快速知道哪些字段需要看，哪些字段已经确认过。
- 不需要在邮件、附件、客户补料链接之间反复切换。
- 有冲突的字段会被突出提示，降低漏审风险。
- 可以直接引用候选值，减少复制粘贴错误。

对业务方：

- 提高制单准确率。
- 降低因字段错填导致的改单、延误和客户投诉。
- 出现问题时能追溯字段从哪里来、何时被改、谁改过。

对平台：

- 字段值不再只是一个最终结果，而是有来源、有状态、有历史。
- 为后续自动审核、客户规则、风险提示和质量分析打基础。
- 可以量化模型抽取质量、客户回填质量和人工修改频率。

## 7. Solution

### 7.1 产品判断

当前设计方向是合理的，但需要明确职责边界：

| 承载对象 | 建议承载内容 | 不建议承载内容 | 理由 |
|---|---|---|---|
| 字段前 icon | 来源入口、确认状态、查看候选值和变更记录 | 不直接承载冲突本身 | icon 面积小，适合表示状态和入口，不适合承载复杂风险 |
| 字段本体 | 当前值、冲突状态、错误状态、必填缺失状态 | 不承载所有来源详情 | 字段本体最接近用户填写和审核动作，适合用颜色提示风险 |
| 候选值弹层 | 所有来源候选值、当前值、已引用、确认状态、变更历史 | 不直接替代字段编辑 | 弹层用于对比和选择，最终值仍回到字段 |

结论：

- icon 承载“已确认/未确认”和“打开候选值/变更记录”是合理的。
- 字段承载“是否冲突、是否错误、是否缺失”是合理的。
- 冲突不应只靠 icon 提示，必须体现在字段名称或字段框上。
- 点击字段或字段冲突态时，应进入候选值弹层，让操作员处理冲突。

### 7.2 UX/Prototypes

#### 7.2.1 字段前 icon

每个可追溯字段前展示一个来源 icon。

icon 状态：

| 状态 | 展示 | 含义 | 触发条件 |
|---|---|---|---|
| 未确认 | 灰色 icon | 字段有当前值，但操作员尚未审核确认 | 初始抽取、客户链接回填、系统自动填入 |
| 已确认 | 绿色 icon 或带勾状态 | 操作员已经审核过该字段 | 字段获得焦点后失焦，或操作员引用候选值，或人工保存确认 |
| 有候选值 | icon 可点击 | 可查看该字段所有候选来源和值 | 字段存在来源记录 |
| 无来源 | 不展示 icon 或展示弱化状态 | 字段没有来源记录 | 纯手工新增且未保存来源时 |

确认规则：

1. 字段默认未确认。
2. 操作员点击或 Tab 聚焦字段后，再失焦，视为已人工审核。
3. 操作员从候选值中点击“引用”，视为已确认。
4. 操作员手动修改字段并保存，保存成功后视为已确认，并生成一条人工修改记录。
5. 客户链接回填的值进入字段时，字段默认未确认，必须由操作员审核。

#### 7.2.2 候选值与变更记录弹层

点击字段前 icon，打开该字段的候选值弹层。

弹层建议展示：

| 字段 | 说明 |
|---|---|
| 来源类型 | 附件、邮件正文、客户链接、人工修改、系统默认 |
| 来源说明 | 文件名、邮件主题、链接提交人、操作人等 |
| 候选值 | 该来源提供的字段值 |
| 取值时间 | 抽取、提交或保存时间 |
| 是否当前值 | 当前字段展示值是否来自该候选值 |
| 是否已引用 | 操作员是否引用过该候选值 |
| 是否已确认 | 当前字段是否已被操作员确认 |
| 操作 | 引用、查看原文、查看历史 |

排序建议：

1. 当前值排第一。
2. 最新客户链接回填值排在较前。
3. 最新邮件或附件解析结果排在较前。
4. 人工修改记录按保存时间倒序展示。

#### 7.2.3 字段冲突态

当字段存在冲突时，字段名和字段框标红。

交互规则：

1. 点击冲突字段名或字段框，打开候选值弹层。
2. 弹层中展示冲突来源和候选值。
3. 操作员点击“引用”后，字段值更新为该候选值。
4. 引用后字段变为已确认。
5. 如果仍存在其他不同候选值，字段不再标红，但弹层中仍保留历史候选记录。
6. 如果操作员直接手动修改并保存，字段变为已确认，人工修改值进入候选值记录。

### 7.3 Key Features

#### 7.3.1 来源类型

系统支持以下字段来源：

| 来源类型 | 说明 | 示例 |
|---|---|---|
| attachment | 附件识别结果 | PDF、Excel、图片、Word |
| email_body | 邮件正文识别结果 | 客户邮件正文 |
| customer_link | 客户链接回填 | 客户通过补料链接提交 |
| manual_save | 人工修改保存 | 操作员保存后的字段值 |
| system_default | 系统推荐默认值 | 船司、单据类型、客户规则默认值 |

#### 7.3.2 字段确认状态

建议字段增加确认状态：

| 字段 | 类型 | 说明 |
|---|---|---|
| confirmed | boolean | 字段是否已被操作员确认 |
| confirmedBy | string | 确认人 |
| confirmedAt | datetime | 确认时间 |
| confirmMethod | enum | focus_blur、quote_candidate、manual_save、bulk_confirm |

确认状态不等于字段无冲突。字段可以先有冲突，操作员处理后变为已确认。

#### 7.3.3 候选值数据结构

建议每个字段维护候选值列表：

| 字段 | 类型 | 说明 |
|---|---|---|
| fieldKey | string | 字段编码 |
| fieldName | string | 字段展示名 |
| value | string | 候选值 |
| normalizedValue | string | 标准化后的值，用于冲突判断 |
| sourceType | enum | 来源类型 |
| sourceId | string | 来源对象 ID，如邮件 ID、附件 ID、链接提交 ID、保存记录 ID |
| sourceLabel | string | 前端展示来源说明 |
| extractedAt | datetime | 抽取或提交时间 |
| isCurrent | boolean | 是否当前字段值 |
| isQuoted | boolean | 是否被引用过 |
| operator | string | 人工修改或确认人 |

#### 7.3.4 冲突定义

字段冲突是指同一个字段存在多个有效来源值，且标准化后不一致。

冲突场景：

| 场景 | 是否冲突 | 处理 |
|---|---|---|
| 新邮件附件值与邮件正文值不一致 | 是 | 字段标红，展示两个来源 |
| 客户链接回填值与当前值不一致 | 是 | 字段标红，展示客户链接和当前值来源 |
| 客户链接、附件、邮件正文三者不一致 | 是 | 字段标红，展示全部候选值 |
| 客户链接值为空，当前字段也为空 | 否 | 字段保持未确认，按必填规则提示 |
| 客户链接值为空，当前字段非空 | 视字段规则而定 | 默认不覆盖当前值；弹层展示为空候选值 |
| 客户链接值非空，当前字段为空 | 否 | 可填入字段，但字段未确认 |
| 人工修改保存值与原抽取值不一致 | 否 | 不视为待处理冲突，作为人工修改记录保留 |
| 新邮件再次解析值与已确认当前值不一致 | 是 | 字段重新标红，确认状态回到未确认或待复核 |
| 系统默认值与人工值不一致 | 否 | 默认不作为冲突，除非该字段配置为需要比较默认值 |

标准化规则建议：

- 去除首尾空格。
- 连续空格合并为一个空格。
- 英文大小写按字段配置决定是否敏感。
- 邮箱、港口、箱号、提单号等字段应使用各自字段规则标准化。
- 空值、`N/A`、`NONE`、`无` 是否等价，需要按字段配置。

#### 7.3.5 客户链接回填规则

客户链接回填进入系统后：

1. 生成 `customer_link` 来源候选值。
2. 如果字段当前为空，客户链接值可自动填入字段，但字段为未确认。
3. 如果字段当前非空且与客户链接值不同，字段标红为冲突。
4. 如果字段当前非空且与客户链接值相同，可保留当前值，并追加来源记录。
5. 如果客户链接值为空，不应自动清空当前字段。
6. 客户链接来源需要记录提交人、提交时间和链接 ID。

#### 7.3.6 往来邮件人工引用规则

当前往来邮件尚未解析到对应提单号，因此不能自动把往来邮件内容写入右侧表单。

规则：

1. 往来邮件解析出的候选值先进入候选池。
2. 操作员需要手动引用到右侧字段。
3. 引用后字段值更新，并标记为已确认。
4. 来源记录显示为邮件正文或邮件附件，并保留邮件主题、发件人、接收时间。

#### 7.3.7 人工修改保存记录

每次操作员修改字段并保存成功后，系统新增一条 `manual_save` 来源记录。

记录内容：

- 字段名。
- 保存前值。
- 保存后值。
- 操作人。
- 保存时间。
- 是否覆盖了冲突值。
- 保存前字段是否已确认。

人工修改保存后：

- 字段当前值更新为保存后的值。
- 字段标记为已确认。
- 原候选值仍保留。
- 后续新来源值与人工确认值不一致时，字段重新进入冲突态。

### 7.4 Technology

前端：

- 字段组件需要支持 icon、确认态、冲突态、候选值弹层。
- 字段聚焦和失焦事件需要触发确认状态更新。
- 点击 icon 或冲突字段时打开候选值弹层。
- 引用候选值后更新字段值、确认状态和冲突态。
- 未确认字段、冲突字段、必填缺失字段需要有不同视觉层级。

后端：

- 字段值需要保存当前值、确认状态和候选值列表。
- 抽取服务需要按字段写入候选值，而不是只写最终值。
- 客户链接提交需要写入候选值来源。
- 保存接口需要生成 `manual_save` 记录。
- 冲突判断应尽量在后端或统一服务中完成，前端只展示结果。

建议接口返回：

```json
{
  "fieldKey": "shipper",
  "fieldName": "Shipper",
  "currentValue": "BEI JING KANG JIE LONG INTERNATIONAL",
  "confirmed": false,
  "conflict": true,
  "candidates": [
    {
      "sourceType": "attachment",
      "sourceLabel": "附件：补料.pdf",
      "value": "BEI JING KANG JIE LONG INTERNATIONAL",
      "isCurrent": true,
      "isQuoted": true,
      "extractedAt": "2026-01-02 10:00:00"
    },
    {
      "sourceType": "email_body",
      "sourceLabel": "邮件正文",
      "value": "BEIJING KANGJIELONG INTL",
      "isCurrent": false,
      "isQuoted": false,
      "extractedAt": "2026-01-02 10:01:00"
    }
  ]
}
```

### 7.5 Assumptions

- 操作员愿意通过字段级确认来完成审核。
- 字段聚焦后失焦可以代表“已看过”，但不一定代表“正确”。后续可增加更强确认动作。
- 不同字段的标准化规则不同，需要逐步配置。
- 客户链接回填内容不一定可信，仍需要操作员确认。
- 往来邮件暂时不能自动匹配提单号，因此先支持人工引用。

## 8. Release

### V1 范围

- 字段前 icon 展示未确认/已确认状态。
- 点击 icon 打开候选值和变更记录弹层。
- 支持附件、邮件正文、客户链接、人工修改、系统默认五类来源。
- 字段冲突时字段名和字段框标红。
- 点击冲突字段可查看来源并引用候选值。
- 字段聚焦后失焦标记为已确认。
- 人工保存生成字段变更记录。
- 客户链接值进入候选值；非空且字段为空时可填入但未确认。

### V2 范围

- 字段级标准化规则配置。
- 批量确认无冲突字段。
- 展示来源原文定位，例如附件页码、邮件片段。
- 支持客户规则来源。
- 支持字段审核统计：未确认数、冲突数、人工修改数。

### V3 范围

- 自动识别往来邮件对应提单号。
- 基于历史人工确认记录推荐可信来源。
- 对高风险字段做强确认，例如提单号、港口、收发货人、箱封号。
- 按客户、船司、字段维度分析模型抽取质量。

## 验收标准

1. 字段默认展示未确认 icon。
2. 字段聚焦后失焦，icon 变为已确认状态。
3. 点击 icon 可以查看该字段全部候选值和变更记录。
4. 附件和邮件正文同字段值不一致时，字段标红。
5. 客户链接值与当前值不一致时，字段标红。
6. 客户链接值非空且当前字段为空时，可以填入字段，但字段仍未确认。
7. 点击候选值“引用”后，字段值更新，字段变为已确认。
8. 人工修改并保存后，生成一条人工修改来源记录。
9. 后续新来源值与已确认当前值不一致时，字段重新标红并进入待复核。
10. 字段冲突被处理后，字段不再标红，但历史候选值仍可查看。

## 待确认问题

1. “字段聚焦后失焦即确认”是否足够，还是需要显式确认按钮。
2. 哪些字段必须强确认，例如提单号、港口、收发货人、箱号、封条。
3. 空值、`N/A`、`NONE`、`无` 在不同字段中是否视为等价。
4. 客户链接空值是否应该作为候选值展示，还是只记录在提交记录中。
5. 人工修改保存记录是否需要展示保存前值和保存后值。
6. 冲突处理后，是否需要保留“曾经冲突”的弱提示。
7. 未确认字段是否允许提交船司，还是只在高风险字段上拦截。
