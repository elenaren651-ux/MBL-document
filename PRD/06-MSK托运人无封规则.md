# 【MSK】MBL制单-托运人无封规则

## 1. Summary

本需求用于解决 MSK 场景下“托运人封条可为空”与“封条漏填”无法区分的问题。平台侧新增封柜类型“托运人无封”，当操作员选择该类型时，封条字段自动清空并置灰，系统允许保存；提交通道时仍映射到托运人封号字段，并传空值。

## 2. Contacts

| 角色 | 关注点 | 备注 |
|---|---|---|
| 业务方 | MSK 托运人封条真实无值时可正常流转 | 需要区分真实无封与漏填写 |
| 平台测试 | 通过通道测试验证传值规则 | 需要明确保存与提交船司的数据规则 |
| 通道侧 | 接收平台传参并提交船司 | 需确认托运人封号字段允许空值 |
| 操作员 | 在制单详情中判断和处理封条信息 | 需要清晰的界面状态 |
| 产品/研发 | 实现校验、保存、提交映射和日志 | 需保持全链路规则一致 |

## 3. Background

当前 MSK 的托运人封条在真实业务中可以为空，但平台制单详情中封条字段目前按必填处理。若原文档没有解析出封条值，操作员无法判断这是客户真实没有提供托运人封条，还是系统解析失败或人工漏填。

业务方希望增加一个明确的封柜类型：“托运人无封”。该类型代表操作员已经确认该箱没有托运人封条，不是漏填。选择该类型后，封条输入框自动置灰，保存时不再校验封条必填。提交给通道侧时，仍按通道契约映射为托运人封号，并传空值。

## 4. Objective

目标是让 MSK 托运人封条为空的业务单据可以被明确标记、保存和提交，同时避免普通漏填被误放行。

关键结果：

- MSK 托运人无封场景下，操作员可以保存制单信息，保存成功率达到 100%。
- 因托运人封条为空导致的误拦截减少到 0。
- 所有选择“托运人无封”的箱信息，在提交通道时托运人封号字段传空值。
- 操作员可以在详情页清楚看到该箱是“托运人无封”，而不是未填写。

## 5. Market Segment(s)

本需求面向处理 MSK MBL 制单的操作员、平台测试和通道侧对接人员。

适用范围：

- 船司：MSK。
- 页面：MBL 制单详情页、待提交船司、提交船司前校验、保存制单信息。
- 数据对象：集装箱信息中的封柜类型、封条/铅封号。

暂不纳入范围：

- 非 MSK 船司是否支持托运人无封。
- 承运人封条、海关封条、兽医封条为空的放行规则。
- 通道侧新增枚举或改造船司官网填报逻辑。

## 6. Value Proposition(s)

对操作员：

- 可以明确表达“该箱确实没有托运人封条”。
- 不需要为了保存而填入无业务意义的占位字符。
- 可减少因为封条为空反复确认、退回和沟通。

对业务方：

- 真实无封业务可以正常推进。
- 漏填与真实无封不再混在一起。
- 提交船司的数据更符合业务实际。

对平台和通道：

- 平台保留清晰的业务状态。
- 通道仍接收原有字段，不强依赖新增枚举。
- 后续排查时可以看到是操作员选择了“托运人无封”，不是系统丢值。

## 7. Solution

### 7.1 UX/Prototypes

在集装箱信息表格的“封柜类型”下拉中，为 MSK 增加选项：

- 承运人封条
- 海关封条
- 托运人封条
- 托运人无封
- 兽医封条

交互规则：

1. 当封柜类型选择“托运人无封”时：
   - 当前行的“封条/铅封号”字段自动清空。
   - “封条/铅封号”字段置灰且不可编辑。
   - 字段可展示占位文案“无封，无需填写”。
2. 当封柜类型从“托运人无封”切换为其他类型时：
   - “封条/铅封号”字段恢复可编辑。
   - 若封条为空，保存时按对应封柜类型的必填规则校验。
3. 已保存或重新进入详情页时：
   - 若该行封柜类型为“托运人无封”，封条字段仍为空且置灰。
   - 不展示封条为空的错误态。

### 7.2 Key Features

#### 7.2.1 封柜类型新增“托运人无封”

系统在 MSK 集装箱信息中支持“托运人无封”类型。该类型表示操作员已确认没有托运人封条。

建议内部枚举：

| 字段 | 值 | 展示文案 | 说明 |
|---|---|---|---|
| sealType | SHIPPER_NO_SEAL | 托运人无封 | 平台内部使用 |
| sealNo | 空字符串或 null | 空 | 不需要填写封条号 |

#### 7.2.2 保存校验规则

保存制单信息时，按以下规则处理：

| 船司 | 封柜类型 | 封条/铅封号 | 是否允许保存 |
|---|---|---|---|
| MSK | 托运人无封 | 空 | 允许 |
| MSK | 托运人无封 | 非空 | 前端应自动清空后保存 |
| MSK | 托运人封条 | 空 | 不允许，提示补充封条 |
| MSK | 承运人封条/海关封条/兽医封条 | 空 | 按现有必填规则处理 |
| 非 MSK | 任意类型 | 空 | 按现有规则处理 |

提示文案建议：

- 当选择“托运人无封”时，不提示错误。
- 当选择“托运人封条”且封条为空时，提示“请填写托运人封条，若业务确认为无封，请选择托运人无封”。

#### 7.2.3 原文档解析为空的处理

当系统从原文档未解析出托运人封条时，不自动判断为“托运人无封”。系统仍展示为空，由操作员确认后手动选择“托运人无封”。

原因：

- 解析为空可能是源文件确实无值。
- 解析为空也可能是解析失败、识别漏掉、源文件格式异常。
- 需要保留人工确认动作，避免自动放行漏填。

#### 7.2.4 提交通道映射规则

选择“托运人无封”时，提交给通道侧的数据规则如下：

| 平台字段 | 平台值 | 通道字段 | 通道值 |
|---|---|---|---|
| 封柜类型 | 托运人无封 | 托运人封号/shipperSealNo | 空字符串或 null |
| 封条/铅封号 | 空 | 托运人封号/shipperSealNo | 空字符串或 null |

推荐原则：

- 平台内部保存“托运人无封”状态。
- 通道侧如果只支持“托运人封号”字段，则传空值。
- 不把“托运人无封”作为封号文本传给通道。
- 不用占位值，如 N/A、无、NONE，除非船司或通道契约明确要求。

#### 7.2.5 提交船司前校验

提交船司前校验需要复用保存校验规则：

- MSK + 托运人无封 + 封条为空：允许提交。
- MSK + 托运人封条 + 封条为空：不允许提交。
- 其他类型按现有规则。

#### 7.2.6 操作日志

当操作员将封柜类型改为“托运人无封”时，记录操作日志。

日志内容建议：

- 操作类型：修改集装箱封柜类型。
- 操作内容：箱号、原封柜类型、原封条、新封柜类型、新封条。
- 操作人、操作时间。

### 7.3 Technology

前端：

- MSK 的封柜类型下拉增加“托运人无封”。
- 选择该类型时清空并禁用封条输入框。
- 切换其他类型时恢复输入框。
- 保存前做行级校验。

后端：

- 数据模型支持保存“托运人无封”。
- 保存接口允许 MSK + 托运人无封 + 封条为空。
- 提交船司接口映射通道参数时，托运人封号传空值。
- 日志记录变更前后的封柜类型和封条值。

通道：

- 确认托运人封号字段支持空字符串或 null。
- 若通道侧对空值有格式要求，以通道契约为准。

### 7.4 Assumptions

- MSK 船司允许托运人封条为空。
- 通道侧可以接收托运人封号为空。
- “托运人无封”只用于平台侧表达已确认无封，不要求船司官网或通道侧新增同名选项。
- 操作员有能力根据原文档或客户信息判断是否为真实无封。

## 8. Release

### V1 范围

- MSK 集装箱信息封柜类型新增“托运人无封”。
- 选择后封条字段自动清空、置灰、不可编辑。
- 保存和提交船司前校验允许该类型封条为空。
- 提交通道时映射托运人封号为空值。
- 记录操作日志。

### V2 可选优化

- 在解析来源中展示“原文档未解析出封条”，辅助操作员判断。
- 对“托运人无封”增加二次确认或备注，适用于高风险客户。
- 统计“托运人无封”使用次数，用于后续判断是否扩展到其他船司。

## 验收标准

1. MSK 单据中，操作员可以在封柜类型下拉选择“托运人无封”。
2. 选择“托运人无封”后，当前行封条字段被清空、置灰、不可编辑。
3. “托运人无封”状态下，封条为空可以保存成功。
4. “托运人封条”状态下，封条为空不能保存，并提示操作员填写或选择“托运人无封”。
5. 保存后重新打开详情页，“托运人无封”状态和封条置灰状态保持一致。
6. 提交船司前，MSK + 托运人无封 + 封条为空可以通过校验。
7. 传给通道侧时，平台将该行映射为托运人封号字段，字段值为空。
8. 通道报文中不出现“托运人无封”作为封号值。
9. 修改为“托运人无封”时，系统记录操作日志。

## 待确认问题

1. 通道侧空值使用空字符串还是 null。
2. “托运人无封”是否只对 MSK 生效，还是后续其他船司也可配置。
3. 若一票多箱中只有部分箱选择“托运人无封”，通道侧是否支持逐箱传空值。
4. 是否需要在提交快照中展示“托运人无封”，方便后续追溯。
